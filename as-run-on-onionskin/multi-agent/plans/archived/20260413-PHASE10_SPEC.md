## What Phase 10 is

Phase 10 is now a cleanup-only closeout phase.

Its purpose is to remove the deprecated shared-output local optimum from the active
architecture, preserve the valid cross-pipeline contract work already landed in 10.1 and 10.2,
and leave the repo ready to enter a forward architecture phase without carrying stale ownership
assumptions.

This phase begins after formal Phase 9 closeout and after the initial Phase 10 contract work.
It does not exist to add new analysis capabilities. It exists to correct planning, ownership,
and migration assumptions before forward architecture resumes.

---

## Core design rules

1. Phase 10 allows only migration, ownership correction, planning-surface correction, and removal of deprecated structures.
2. Phase 10 does not allow new analysis capabilities, completeness work, or forward-facing feature work.
3. Pipeline-derived outputs belong under the pipeline that computes them.
4. Shared code reuse and shared emitted-file ownership are separate questions; only the former remains part of the target architecture.
5. Controller/run-level artifacts must remain analysis-neutral.
6. Cross-pipeline synthesis remains a later explicit layer above pipeline-local products.
7. `--pipelines` combinations remain product contracts, not best-effort conveniences.
8. Input capability must be defined from the full sample structure, not just a legacy mode label:
   distinct stage count, replicate count per stage, and denominator/reference availability all matter.
9. `multistage` remains a property of input data, not a synonym for the growth-model pipeline.

---

## Non-goals

- Do not implement new analysis capabilities during Phase 10.
- Do not do pipeline-completeness work during Phase 10.
- Do not add forward-facing features during Phase 10.
- Do not retain or expand any deprecated shared emitted-output layer during Phase 10.
- Do not use migration convenience as a reason to preserve deprecated ownership patterns.

## Priority 10.1 — Cross-pipeline contract audit

### Implementation status

Completed in `v0.10.00`.

### Findings

- HMM had retained private bedGraph sequence-scan/filter helpers rather than reusing a shared contract surface.
- Mixed multistage+HMM runs did not share one effective filtered sequence universe: HMM honored `--min-seq-length` and `--min-bin-count-per-seq`, while the multistage/per-stage lane only saw the raw chromosome selection/default.
- The posterior branch invoked the HMM engine twice; the second invocation fell back to the original manifest instead of the posterior manifest.
- `--pipelines all` still excludes `hmm`; this remains an explicit deferred contract decision for Priority 10.5 rather than something to flip silently during the audit patch.

### Goal

Map the current behavior of `growth`, `per-stage`, and `hmm` with respect to manifest/sample
loading, input filtering, shared intermediate files, and output-layout assumptions.

### Questions this audit must answer

- Which loaders/readers/preprocessors are still duplicated or privately reimplemented?
- Which outputs are truly pipeline-derived versus truly analysis-neutral run metadata?
- Which flags already behave as run-level contracts, and which still behave differently by
  pipeline arm?
- Which current output-path assumptions would block clean pipeline-local ownership and
  analysis-neutral controller metadata?

### Deliverable

A concrete contract matrix and implementation queue, not vague cleanup notes.

## Priority 10.2 — Shared filtering semantics

### Implementation status

Completed in `v0.10.20`.

### Goal

Define and enforce one shared interpretation for:

- `--chromosomes`
- `--min-seq-length`
- `--min-bin-count-per-seq`

### Required outcome

In a multi-pipeline run, the requested input universe is chosen once and then used by every
pipeline arm unless a documented exception is biologically justified.

The controller contract should also state what each pipeline is allowed to do for a given input
structure. That structure must include:

- number of distinct stages
- replicate count per stage
- whether a denominator/reference stage is explicitly available

This is a better long-term contract surface than the older `single-stage` versus `multistage`
split.

### Implementation expectation

Filtering should move conceptually above the individual pipeline arms rather than remaining an
HMM-only visible preprocessing quirk.

### Closeout note (`v0.10.20`)

- The original 10.2 aim was shared filtering semantics for `--chromosomes`, `--min-seq-length`, and `--min-bin-count-per-seq`.
- During implementation, nearby controller and manifest/normalization side quests were discovered and fixed because they blocked or distorted that shared-contract surface.
- Priority 10.2 is now considered complete. Later APS completion, posterior independence, and cross-pipeline synthesis questions are explicitly deferred to later priorities rather than kept here as spillover.

### Clarified HMM preprocessing direction

- Ratio normalization is a supported HMM preprocessing branch, not an architectural requirement.
- Single-file and true single-stage runs without a denominator should support a no-ratio HMM path:
  - Current plan: `step1 → step2 → modified step3 for removing bins that span assembly gaps (but not all "zero bins") → modified step4 chromosome-median normalization on the combined track`.
    - Deprecated earlier plan: `step1 → step2 → skip step3 → modified step4 chromosome-median normalization on the combined track`.
- The no-ratio HMM path is also an intended user option even when multiple stages are provided.
- Reference-backed two-stage inputs can continue to use the current ratio-normalized HMM path.
- The no-ratio branch must document its increased exposure to collapsed repeats and its intended mitigation via the existing triangle-vs-flat / non-growth filtering machinery.
- The no-ratio branch can benefit from two onionskin modules already in place: (1) missingness / gap detection, and (2) triangle vs flat shape BIC scoring for identification of candidate collapsed repeat regions.


### Delivered in first patch

- Shared bedGraph sequence-stat/filter helpers now live in `onionskin_core.io` and are reused by the HMM engine.
- Mixed multistage+HMM runs now resolve one effective chromosome set before dispatch and pass that same set into the multistage/per-stage and HMM lanes.
- Posterior reruns now reuse that same resolved sequence set.
- Manifest-backed single-stage runs and `--onionskin` mixed runs now route into the HMM engine through the shared controller path.
- One-group HMM runs were first routed into a no-ratio preprocessing branch instead of forcing denominator-based ratio normalization.
- Synthetic manifests written for `--onionskin` HMM runs now use the shared manifest format.

### Delivered controller step in `v0.10.03`

- `decide_mode()` is now capability-aware rather than purely stage-count-driven.
- Two-stage inputs with `--norm-mode ref-stage` and an explicit in-manifest `--ref-stage` now stay on the reference-backed single-stage controller.
- Two-stage inputs without that explicit reference-backed normalization contract now route to the growth controller instead of being forced onto the old `<=2 stages => single-stage` path.
- Stage-structured inputs with singleton replicate stages now emit an explicit warning about sparse replicate support.
- This is the first controller patch in the new direction, not the final admissibility matrix.

### Delivered controller step in `v0.10.04`

- Requested pipelines are now resolved against the current input capability contract instead of being interpreted only from the raw `--pipelines` string.
- One-stage and reference-backed two-stage inputs now reject explicit `growth` requests cleanly instead of silently tolerating them.
- Two-stage non-reference-backed inputs can still route to the growth controller, but `--pipelines per-stage` now stays on the per-stage controller for those inputs rather than unnecessarily invoking the growth controller.
- `--pipelines per-stage` on 3+ stage inputs now errors explicitly as a currently unsupported standalone controller combination instead of silently running the growth controller underneath it.

### Delivered controller step in `v0.10.05`

- `--pipelines per-stage` on 3+ stage inputs is now a real standalone controller path rather than an error or a hidden dependency on the growth controller.
- The top-level controller now reuses the same stage-structured manifest loading and normalization logic that the growth engine previously used before dispatching `run_per_stage_mean_shift()` as a side-pipeline.
- Two-stage non-reference-backed `--pipelines per-stage` also now uses that standalone per-stage path, while two-stage requests that include `growth` still route through the growth controller.
- `--posterior` remains unsupported for standalone multistage per-stage runs because the current posterior regrouping path still depends on the growth/APS lane.


### Delivered in second patch (`v0.10.07`)

- The no-ratio HMM branch no longer skips step 3. It now runs a dedicated gap-removal step 3 before chromosome-median normalization.
- The gap-removal source comes from the existing gap-mask logic in `onionskin_core.timing`, but the HMM path now uses it conservatively: a bin is removed only when it is zero-valued across all step-2 group-median bedGraphs. This keeps the no-ratio branch focused on likely assembly gaps rather than file-specific zero bins.
- The above gap-removal rule now applies to chromosome-median HMM preprocessing generally, not just to one-group HMM.
- Stage numbering is no longer treated as arbitrary. Manifest stages are now validated against onionskin's canonical `1..N` developmental-order contract, and non-canonical manifests now fail explicitly instead of being silently renumbered.
- Synthetic manifests produced from convenience inputs now emit at the grouping level (`01-prior/00-synthetic-manifest.tsv` or `02-posterior/00-synthetic-manifest.tsv`) rather than being hidden only inside the grouped HMM directory.
- Synthetic manifests are now written in canonical stage order, and the `--onionskin/--reference` convenience path now reserves stage 1 for reference and stage 2 for onionskin; non-default `--ref-stage` requests on that convenience path now fail explicitly.
- `--ref-stage` is now presented as an advanced override in the CLI help, with text emphasizing the normal onionskin expectation that stage 1 is the earliest pre-amplification reference.

### Delivered in third patch (`v0.10.08`)

- The top-level CLI now exposes pipeline-specific normalization overrides: `--growth-norm-mode`, `--per-stage-norm-mode`, and `--hmm-norm-mode`.
- These new flags currently default to `inherit`, which preserves the existing global `--norm-mode` behavior unless a pipeline-specific override is explicitly requested.
- The controller contract is no longer forced to read only the shared `--norm-mode` value. It now resolves a controller-driving normalization mode from the requested pipelines so that a pure HMM run, a pure per-stage run, and a growth run can ask for different normalization semantics without pretending they are all the same product contract.
- HMM preprocessing is no longer tied only to the number of detected groups. The HMM engine now accepts an explicit preprocessing normalization mode, which makes two-stage HMM runs able to request chromosome-median no-ratio preprocessing directly instead of being forced onto the ratio-backed branch.
- The current split is still intentionally conservative: growth and per-stage cannot yet request divergent normalization modes inside the same combined run because they still share one stage-structured controller path when run together. That conflict now fails explicitly instead of being silently flattened.
- Explicit `ref-stage` overrides for HMM and per-stage now fail cleanly when only one stage is present.

### Delivered in fourth patch (`v0.10.09`)

- `onionskin.py` no longer relies on the multistage engine's hidden per-stage side-pipeline when both `growth` and `per-stage` are requested together.
- In combined growth+per-stage runs, the growth-model lane now runs first through the multistage engine with `--skip-per-stage`, and the per-stage lane is then run independently through the standalone per-stage controller.
- This uncouples per-stage from the growth lane at the top-level controller, which in turn allows divergent growth-vs-per-stage normalization modes to run cleanly in the same combined run.
- The current uncoupling is controller-level, not yet full downstream independence: posterior regrouping is still driven by the growth/APS lane, so the standalone per-stage posterior gap remains a later 10.4 issue.
- The top-level help output was also updated to match the continuum framing better: the old `Single-sample specific` and `Multistage specific` headings are now replaced with `Per-stage / single-stage detection` and `Growth-model stage-structured`.

### Delivered in fifth patch (`v0.10.10`)

- The top-level normalization overrides now have real stage-aware default behavior instead of acting only as `inherit` shims.
- Growth now defaults to `chrom-median` regardless of stage count.
- HMM now defaults to `ref-stage` when 2+ stages are present with the configured reference stage, and otherwise falls back to the shared global `--norm-mode`.
- Per-stage now follows the same stage-aware default as HMM at the controller level: `ref-stage` when 2+ stages are present with the configured reference stage, otherwise the shared global `--norm-mode`.
- The explicit `inherit` option remains as a compatibility escape hatch for users who want a pipeline to follow the global `--norm-mode` even when the new pipeline-specific default would differ.
- The top-level help text for `--growth-norm-mode`, `--per-stage-norm-mode`, and `--hmm-norm-mode` now documents those actual defaults instead of claiming they all inherit by default.

### Delivered in sixth patch (`v0.10.11`)

- The growth engine no longer exposes or consumes the legacy `--skip-per-stage` flag.
- The growth engine also no longer owns a hidden per-stage side-pipeline or its per-stage-specific tuning flags; pipeline composition is now controller-owned rather than engine-owned.
- Combined growth+per-stage runs still work through `onionskin.py`, but the sequencing now reflects controller orchestration only. The growth engine itself is once again just the growth-model lane implementation.
- This is the first concrete implementation step toward a real dependency-graph controller where `--pipelines` is the only user-facing pipeline-selection surface and cross-pipeline synthesis happens downstream of independent pipeline arms.

### Delivered in seventh patch (`v0.10.12`)

- The 3+ stage standalone `per-stage` default was narrowed back to the older shared-global behavior instead of automatically switching to `ref-stage` whenever a reference stage exists.
- Current contract after this correction:
  - growth defaults to `chrom-median`
  - HMM defaults to `ref-stage` whenever 2+ stages are present with the configured reference stage
  - per-stage defaults to `ref-stage` only for two-stage reference-backed inputs; on 3+ stage inputs it follows the shared global `--norm-mode` unless explicitly overridden
- This matches the historical multistage/per-stage behavior more closely and avoids silently imposing a new reference-stage default on broader stage-structured per-stage runs without explicit biological justification.
- Focused runtime tests now verify both the corrected 3+ stage default and the explicit `--per-stage-norm-mode ref-stage` override on real multistage toy manifests.

### Delivered in eighth patch (`v0.10.13`)

- The pipeline-specific normalization overrides now distinguish an unset flag from an explicit `inherit` request.
- Parser defaults for `--growth-norm-mode`, `--per-stage-norm-mode`, and `--hmm-norm-mode` are now suppressed so the controller can treat an absent flag as "use the pipeline-specific runtime default" and an explicit `inherit` token as "follow the shared global `--norm-mode`".
- Growth keeps the same effective default (`chrom-median`), but the contract is now real instead of being conflated with the parser default token.
- Per-stage now has a fully clarified default contract: `chrom-median` for one-stage inputs, `ref-stage` for two-stage reference-backed inputs, and the shared global `--norm-mode` for broader stage-structured inputs unless the user explicitly overrides it.
- HMM now has a fully clarified default contract: `chrom-median` for one-stage inputs, `ref-stage` for reference-backed 2+ stage inputs, and otherwise the shared global `--norm-mode` unless the user explicitly overrides it.
- The CLI help text now reflects those runtime semantics directly and no longer prints the misleading argparse `(default: inherit)` suffix for the per-pipeline normalization overrides.
- The `--norm-mode` help text is also now explicit that the main onionskin APS post-processing pass and HMM step-14 APS are currently separate normalization surfaces.

### Delivered in ninth patch (`v0.10.15`)

- `--force-manifest` is now available as an explicit compatibility path for non-canonical user manifests.
- Strict manifest validation remains the default contract, but forced-manifest runs can deterministically remap arbitrary user stage labels onto onionskin's runtime `1..N` stage contract before any controller or engine dispatch.
- Forced-manifest runs now materialize the canonical runtime manifest visibly at `01-prior/00-synthetic-manifest.tsv` and emit a stage translation table alongside it.
- Hires manifests are remapped in lockstep with the primary manifest, but not by blind coercion: the force path first verifies same row count, same basename order, and the same pre-remap sample-to-group assignment pattern before allowing the hires remap to proceed.
- Posterior remains compatible because the translated prior manifest is then treated like any other canonical runtime manifest; posterior manifest construction continues to operate from the runtime manifest plus the same hires sample-order contract.
- The CLI help text for `--hires-manifest` now makes the same-sample / same-order / matching-filename expectation explicit.

### Delivered in tenth patch (`v0.10.17`)

- Explicit `ref-stage` requests are now validated consistently at the controller boundary rather than failing later and inconsistently inside individual pipeline arms.
- Growth, per-stage, and HMM now all fail cleanly when their resolved normalization contract is `ref-stage` but the configured `--ref-stage` is not actually present in the manifest.
- This closes a real remaining 10.2 normalization/reference-semantics hole: growth and standalone per-stage previously crashed later in `apply_ref_stage_baseline()`, while HMM could proceed further under the same invalid reference-stage contract.
- Focused regression coverage now locks in the clean failure surface for all three active lanes.

### Delivered in eleventh patch (`v0.10.19`)

- The CLI and `PIPELINE_SPEC.md` wording now match the live APS wiring more precisely.
- The main onionskin APS pass is documented as following the resolved growth/per-stage lane normalization contract rather than vaguely implying one separate global APS-only normalization rule.
- HMM step-14 APS remains documented as a separate normalization surface using the resolved HMM contract.
- Forced-manifest hires coverage now explicitly locks in the two remaining simple rejection surfaces that were previously only implied by code: mismatched row count and mismatched basename order.

### Delivered closeout framing in `v0.10.20`

- Priority 10.2 is now explicitly closed back to its original scope: shared filtering semantics plus the nearby controller/manifest contract fixes needed to make those filters behave consistently across pipelines.
- APS is no longer framed here as a growth-owned or onionskin-main product. The intended architecture is pipeline-local APS outputs first, with any later unified APS layer deferred to cross-pipeline synthesis work.
- Pipeline-local posterior grouping, posterior ordering, and posterior manifest products are likewise treated as standalone pipeline-completeness work rather than as 10.2 blockers.
- Cross-pipeline posterior/synthesis questions are now explicitly deferred to Priority 10.5.

### Priority boundary clarification after `v0.10.11`

- `v0.10.11` belongs in Priority 10.2 because removing engine-owned pipeline control is still controller-contract cleanup.
- The heavier dependency-graph follow-through should not be treated as the next immediate 10.2 step just because `v0.10.11` points in that direction.
- The cleaner ordering is:
  1. finish the remaining true 10.2 semantics first;
  2. handle shared stage-structured preparation and ownership work under 10.3;
  3. handle standalone per-stage downstream independence under 10.4;
  4. handle full composition guarantees and explicit dependency-graph semantics under 10.5.

### Deferred after 10.2 closeout

- Cleanup of deprecated ownership assumptions now belongs to Priorities 10.3 through 10.6.
- Pipeline-local completeness now belongs to the Phase 11 forward architecture plan rather than to any remaining Phase 10 work.
- Until cross-pipeline synthesis exists, or when it is intentionally disabled in the future, each pipeline-local posterior in `02-posterior/` should descend from the posterior manifest produced by that same pipeline's `01-prior/` results.
- Cross-pipeline synthesis questions now belong to the future architecture plan rather than to any remaining Phase 10 implementation.
- If later work exposes a true regression in shared filtering or nearby controller semantics for `--chromosomes`, `--min-seq-length`, or `--min-bin-count-per-seq`, reopen 10.2 narrowly rather than using it as a catch-all architecture bucket.

- Any planning note that treated `00-signal-files/` as a retained architectural layer, canonical
