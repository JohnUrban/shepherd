## Phase scope and framing (preamble)

These four BRAIN entries (BRAIN15.1–4) frame the phase rather than defining priorities. They are SPEC preamble.

**BRAIN15.1 — Phase scope:** finishing HMM pipeline completeness and enriching it. HMM should be self-sufficient when run alone — capable of producing its own analyses downstream of amplicon calling (summit detection + refinement, APS, posterior ordering/grouping/manifest, timing, fork tracking, fork age). Increase accuracy and reliability of HMM posterior manifests + summit estimation. Expand analyses, plots, notebooks. HMM-centric but explicitly cross-pipeline-aware: changes that touch controller-level architecture (gap mask, missingness diagnostic) cascade to growth and RMS by inheritance.

**BRAIN15.2 — Core design rules** (in force):
1. Pipeline-derived outputs belong under the pipeline that computes them.
2. Shared computation belongs in modules, not in shared emitted-analysis directories.
3. Controller/run-level artifacts must remain analysis-neutral.
4. No pipeline is the default, canonical, authoritative, or main pipeline.
5. Until explicit synthesis exists, each pipeline remains independent in both `01-prior/` and `02-posterior/`.
6. Cross-pipeline synthesis remains a separate explicit layer above pipeline-local products.
7. Completeness work must respect pipeline-local ownership rather than flattening differences into centralized intermediate files.
8. Any change that alters emitted paths, output schemas, or inspection-relevant analysis surfaces must update the affected utilities, inspectors, and regression helpers in the same change.
9. As new pipeline-local features come online, extend the relevant inspection and regression surfaces.
10. `multistage` remains an input-data property, not the name of the growth pipeline.
11. `--pipelines all` means all three pipelines: `growth`, `per-stage`, and `hmm`.

**BRAIN15.3 — Active carry-over decisions from late Phase 10–11** (settled facts):
- Long-term pipeline term remains `growth-model`; `multistage` remains data/property term.
- Grouped pipeline order: `01-hmm/`, `02-growth-model/`, `03-rcn-mean-shift/`.
- Engine names: `hmm_engine.py`, `growth_model_engine.py`, `rcn_mean_shift_engine.py` (batch), `rcn_mean_shift_singlefile_engine.py` (single-file CLI). Shared RCN helpers in `onionskin_core/rcn_mean_shift_helpers.py`.
- Each pipeline produces its own posterior manifest from its own prior outputs.
- First synthesis target = call-level synthesis across pipeline-local prior outputs; downstream analyses then recompute from the synthesized call set (NOT in Phase 15 scope).
- Capability matrix for pipeline admissibility still needs an explicit written home in the live planning surface (NOT in Phase 15 scope; tracked elsewhere).

**BRAIN15.4 — Non-goals** (do NOT do):
- Do not revive deprecated shared emitted-output ownership patterns.
- Do not approximate synthesis through centralized intermediate files.
- Do not treat growth as the owner of common analysis surfaces.

**BRAIN15.5 — HMM completeness matrix** is the umbrella cross-reference index. Its analysis-families gap-audit table ([PHASE15_BRAINSTORM.md § BRAIN15.5 lines 105–122](PHASE15_BRAINSTORM.md)) is the deliverable checklist for Phase 15. Each row's BRAIN entry is owned by a SPEC priority below; this SPEC's coverage of the table is auditable against that BRAINSTORM section. The matrix itself is preamble, not a standalone priority.

**3. Three PCA layers (orthogonal, composable):**

- **Layer 1 — amplicon-level shape PCA.** For each amplicon separately, fit PCA on the (n_samples × n_bins_in_amplicon) shape sub-matrix; take top X PCs per amplicon. - **Layer 3 — matrix-level PCA.** Generic post-feature-construction projection: whatever feature matrix `--aps-feature` produced (M × any-N), PCA-project to M × X before clustering. Independent toggle that composes with any Layer 1/Layer 2 / metrics mode.

The three layers compose. Example: `--aps-feature shape_pca --aps-pca-matrix --aps-final-pc 2` does Layer 1 (per-amplicon PCA on shape) → Layer 3 (PCA on resulting matrix). Or `--aps-feature metrics --aps-pca-matrix` does the all-metrics composite then Layer-3 PCA-projects before clustering.

**8. Label dictionary internal architecture (NEW deliverable 2026-05-03).** Internal HMM math is 0-based throughout the codebase — single source of truth at the computational layer. **Layout contract (cross-pipeline):**
- **Final pass-2 outputs live at the normal step paths** (top level of each step directory) — downstream consumers always read pass-2 (final) results from the well-known step paths regardless of whether the two-pass flag was on.
- **Pass-1 artifacts are archived under `pass1/` subdirectory of each step directory** when pass-2 begins. The move from top-level → `pass1/` happens atomically at the pass-2 transition. Applies to all three pipelines' step directories where pass-1 outputs were written.
- **Sentinel/metadata file at each affected step directory** records the two-pass provenance: `.pass_metadata.json` (or similar) containing fields like:
  - `two_pass_run: bool` — whether the directory's contents reflect a two-pass run.
  - `pass1_archived_at: <ISO-8601 timestamp>` — when pass-1 outputs moved to `pass1/`.
  - `pass2_completed_at: <ISO-8601 timestamp>` — when pass-2 outputs landed at top level.
  - `--second-pass-background-estimation: <flag value>` — flag value at the time of the run.
  - `pipeline: <growth|rcn-mean-shift|hmm>` — which pipeline this directory belongs to.

**Atomicity safeguard:** the move from top-level → `pass1/` MUST be atomic (or guarded by the sentinel file's state) so an interrupted run doesn't leave a half-archived directory. **Layout contract:**
- **Final pass-2 outputs live at the normal step paths** (top level of each step directory) — downstream consumers always read pass-2 (final) results from the well-known step paths regardless of whether the two-pass flag was on.
- **Pass-1 artifacts are archived under `pass1/` subdirectory of each step directory** when pass-2 begins. The move from top-level → `pass1/` happens atomically at the pass-2 transition.
- **Sentinel/metadata file at each affected step directory** records the two-pass provenance: `.pass_metadata.json` (or similar; SPEC engineering finalizes the filename) containing fields like:
  - `two_pass_run: bool` — whether the directory's contents reflect a two-pass run.
  - `pass1_archived_at: <ISO-8601 timestamp>` — when pass-1 outputs moved to `pass1/`.
  - `pass2_completed_at: <ISO-8601 timestamp>` — when pass-2 outputs landed at top level.
  - `--second-pass-background-estimation: <flag value>` — flag value at the time of the run.
- **Upstream-vs-downstream rule (no silent cramming):** **no later-stage derived file is silently written into an upstream step directory without pass/provenance labeling.** If a step-9+ analysis needs to write back to a step-1–8 directory, it must either (a) write to a clearly-labeled sub-directory with provenance metadata, or (b) write to its own step directory. Silently overwriting upstream files is a contract violation.

**Why option (B) over option (A):** Option (A) requires every code path that reads from a previous step's directory to know which pass it's reading from — significant code-change surface. Option (B) keeps the well-known top-level path as the canonical "final" results path; pass-1 lives in `pass1/` for diagnostics. Per BRAIN15.25 user note recommendation + Codex Finding 5 audit recommendation, option (B) is the locked choice.

**Atomicity safeguard:** the move from top-level → `pass1/` MUST be atomic (or guarded by the sentinel file's state) so an interrupted run doesn't leave a half-archived directory. Recommended implementation: write `.pass_metadata.json` with `pass2_in_progress: true` BEFORE the move; complete the move; complete the pass-2 re-run; finalize `.pass_metadata.json` with `two_pass_run: true` + `pass2_completed_at`. On startup, the engine checks the sentinel: `pass2_in_progress: true` without `pass2_completed_at` means an interrupted run; recover by either completing pass-2 OR rolling back the archive.
