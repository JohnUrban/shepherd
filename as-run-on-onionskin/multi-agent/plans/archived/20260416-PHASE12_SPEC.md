## Priority 12.1 — HMM steps dict migration to `output_layout.py`

**Source:** BRAINSTORM-F.
**Type:** Pure refactor — no HMM step paths change, no behavioral change.
**Purpose:** Establish `output_layout.py` as the single canonical path authority for both
pipelines before Priority 12.2 extends the same pattern to growth.

### Deliverables

#### `onionskin_core/output_layout.py`

Add a new function `build_hmm_steps(hmm_out_dir: str) -> dict`:

```python
def build_hmm_steps(hmm_out_dir: str) -> Dict[str, str]:
    """Return a dict mapping HMM step names to their numbered directories.

    hmm_out_dir is the absolute path to the pipeline's 01-hmm/ container
    (e.g., <out_dir>/01-prior/01-hmm/).
    """
    return {
        "step1":  os.path.join(hmm_out_dir, "01-mednorm"),
        "step2":  os.path.join(hmm_out_dir, "02-unionStats"),
        "step3":  os.path.join(hmm_out_dir, "03-removeZeroBins"),
        "step4":  os.path.join(hmm_out_dir, "04-chromMedRatioNorm-RCN"),
        "step5":  os.path.join(hmm_out_dir, "05-medianSmoothed-RCN"),
        "step6":  os.path.join(hmm_out_dir, "06-HMM"),
        "step7":  os.path.join(hmm_out_dir, "07-collapsedHMM"),
        "step8":  os.path.join(hmm_out_dir, "08-aboveBackground"),
        "step9":  os.path.join(hmm_out_dir, "09-summitStates"),
        "step10": os.path.join(hmm_out_dir, "10-summitBins"),
        "step11": os.path.join(hmm_out_dir, "11-amplicon-metrics"),
        "step12": os.path.join(hmm_out_dir, "12-fork-travel"),
        "step13": os.path.join(hmm_out_dir, "13-summit-refinement"),
        "step14": os.path.join(hmm_out_dir, "14-aps"),
        "step15": os.path.join(hmm_out_dir, "15-timing"),
        "step16": os.path.join(hmm_out_dir, "16-clustering"),
        "notebooks": os.path.join(hmm_out_dir, "notebooks"),
    }
```

**Note:** These step dir names are taken directly from the live `hmm_engine.py` dict and
must not be changed — they control where HMM outputs land. The `"notebooks"` key is
present in the live dict and must be included here.

#### `onionskin_core/engines/hmm_engine.py`

- In `run_hmm()`, **remove** the internal `steps = { ... }` dict construction.
- Change `run_hmm()` signature to accept `steps: Dict[str, str]` as a parameter (passed
  in from the caller in `onionskin.py`).
- All internal references to `steps["stepN"]` remain unchanged — only the construction
  site moves.

#### `onionskin.py`

- In `_run_hmm_engine()`, **before** calling `run_hmm()`:
  - Import and call `build_hmm_steps(hmm_out_dir)` to construct the steps dict.
  - Pass the resulting `steps` dict to `run_hmm()`.
- Ensure `hmm_out_dir` is the same path that was previously passed to or derived inside
  `run_hmm()` so the absolute paths are identical.

#### External scripts (audit pass)

Audit the following scripts for hardcoded HMM step dir names (e.g., `"01-mednorm"`,
`"05-medianSmoothed-RCN"`, `"14-aps"`, etc.) and update them to use the canonical paths
from `build_hmm_steps()` where practical:

- `scripts/generate_hmm_v2_references.py`
- `scripts/compare_puffstep_outputs.py`
- `scripts/summit_inspector.py` (HMM path references only)

If a script reads paths from a live layout dict (e.g., by calling `build_output_layout()`),
it should also call `build_hmm_steps(layout["hmm_dir"])` to get step paths rather than
hardcoding them.

### Validation

After the migration:

1. Run `make test` — all tests must pass.
2. Run `make toy` — verify HMM outputs still land in the correct numbered dirs.
3. Manually confirm that `build_hmm_steps(layout["hmm_dir"])` returns paths identical to
   what `hmm_engine.py` was previously constructing internally. A quick `assert` or debug
   print comparing the two dicts is sufficient during development.

### Scope boundary

This Priority does **not** change any HMM step dir names, any HMM output files, or any
HMM behavior. If any behavioral change is discovered during the migration, stop and flag it
before proceeding.

## Priority 12.2 — Growth pipeline 15-step layout

**Source:** BRAINSTORM-A + BRAINSTORM-D + Copilot [CP-1]–[CP-5].
**Type:** Major structural refactor with mandatory atomic consumer updates.
**Dependency:** Priority 12.1 must be complete (`output_layout.py` is the path authority).

**Key decisions (from brainstorm design session):**

- **No root files** in `02-growth-model/` — all output files in numbered step dirs.
- **`organize_outputs()` is retired** — it only existed because the engine wrote flat.
  Once the engine writes to step dirs directly, remove all `organize_outputs()` call sites
  in `onionskin.py` and delete the function (do not keep as dead code).
- **`--resolve-overlaps` flag is removed** — overlap resolution is always mandatory;
  no use case for skipping it. Remove from argparse and remove all
  `if getattr(args, "resolve_overlaps", False):` guards in `_phase2_postprocess()`.
- **`06-stage-medians/` is kept** — stage-median bedGraphs within called intervals are
  used internally by `resolve_overlaps_bed()` and `write_profile_plots()`. Retaining them
  on disk is consistent with HMM retaining its intermediate step outputs.
- **Posterior manifest files get a formal exception** — `posterior_manifest.tsv` and
  `posterior_hires_manifest_*.tsv` are controller-owned metadata, not growth engine outputs.
  They move **up** to the grouping dir level (`02-posterior/`) rather than living inside the
  growth pipeline container (`02-posterior/02-growth-model/`). This is the only permitted
  exception to the "no root files in growth arm" rule.
- **All consumer updates are mandatory and atomic with the layout change** — exact-path
  consumers that break immediately must be in the same Priority, not deferred.

## Priority 12.4 — Compatibility Matrix (Role 1 audit, investigation-only)

**Auditor:** Claude Sonnet 4.6  
**Version tag:** v0.12.18  
**Verdict:** NOT FEASIBLE in current architecture.

### Compatibility matrix

| Dimension | Finding | Blocker? |
|-----------|---------|----------|
| **1. Smoothing algorithm** | `--hmm-smooth-halfwidth` feeds `local_median_smooth_bedgraph()` (median); `--growth-rcn-smooth-bins` feeds `smooth_mean()` (mean). Same units (halfwidth in bins), same default (3 → 7-bin window), but different algorithms. Reuse would silently change growth smoothing behavior. | Medium — behavioral change, not hard incompatibility |
| **2. Zero-bin / missing-bin semantics** | HMM step-3 removes zero bins from **per-stage group median** bedGraphs before normalization (`remove_zero_bins()` on denominator files). Growth applies inline per-sample masking in the evidence-track loop (lines 1246-1249 of `growth_model_engine.py`): bins where ≥90% of individual samples are zero or non-finite are NaN-masked. These are different operations on different data shapes. | Hard — masking semantics not equivalent |
| **3. Producer shape vs consumer need** | **Hard blocker.** HMM steps 2-5 operate entirely on **per-stage group aggregate** bedGraphs. Step 2 collapses per-sample normalized tracks into one median bedGraph per stage. Steps 3, 4, 5 all process those stage-level aggregates. Step-5 outputs are one smoothed RCN bedGraph per non-reference stage. Growth (`load_tracks_from_manifest()`) reads **per-sample individual bedGraphs** and builds a `{sample_id: {chrom: df}}` matrix — it needs one file per sample, not one file per stage. Step-5 products cannot be substituted without restructuring either the HMM pipeline (emit per-sample step-5 intermediates) or the growth engine (accept pre-aggregated stage tracks). | Hard — fundamental shape mismatch |
| **4. KNOWN_ISSUES ordering conflict** | [ISSUE:2026-04-19:1]: HMM step-14 APS reads raw manifest bedGraphs, not step-5 products. The step-14 normalization inconsistency is a separate known issue orthogonal to this feasibility question. | None — no ordering conflict |

### Hard blockers summary

Two independent hard blockers prevent feasibility:

1. **Shape mismatch (Dimension 3):** HMM step-5 produces per-stage aggregate bedGraphs (one per stage). Growth requires per-sample individual bedGraphs (one per sample). These cannot be substituted directly.

2. **Zero-masking semantic divergence (Dimension 2):** HMM zero-bin removal operates on per-stage group medians at step-3, before normalization. Growth zero-masking operates on the per-sample matrix at evidence-track build time, after `load_tracks_from_manifest()`. Bridging these would require non-trivial redesign.

### Recommendation

**NOT FEASIBLE** in current architecture. Close the investigation.

Long-horizon path (if ever revisited): HMM step-5 would need to also emit per-sample smoothed intermediates (not just group aggregates), growth would need to accept pre-smoothed per-sample bedGraphs instead of raw manifest paths, and both pipelines would need to agree on a single smoothing algorithm. This is a significant architectural change with no clear short-term benefit. Add to BRAINSTORM.md as a long-horizon idea and close Priority 12.4.

### FIND-W1 — repair instructions

**File:** `onionskin_core/readme.py`, lines 51–53.

A posterior manifest at `02-posterior/posterior_manifest.tsv` appears in `all_files` as that full relative path, not as the bare filename `posterior_manifest.tsv`. The three boolean checks on lines 51–53 use bare-filename equality/prefix matching and therefore always return `False` for Phase-12-layout posterior runs.

**Repair:** Replace lines 51–53 with basename-aware checks using `Path(path).name`. `Path` is already imported at the top of the file.

```python
has_posterior_manifest = any(path == "posterior_manifest.tsv" for path in all_files)
```

New (lines 51–53):

```python
has_posterior_manifest = any(Path(path).name == "posterior_manifest.tsv" for path in all_files)
has_posterior_cluster_map = any(Path(path).name == "posterior_manifest_cluster_map.tsv" for path in all_files)
```
