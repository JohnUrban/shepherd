## Overview

Phase 14 is a ground-up cleanup of the onionskin CLI (`build_parser()` in `onionskin.py`).
Goals:
1. Make each flag's pipeline ownership unambiguous from its name alone.
2. Eliminate legacy naming that predates the current growth / hmm / rms vocabulary.
3. Group flags by conceptual category, not implementation accident.
4. Replace silent legacy aliases with explicit deprecation errors.
5. Make `onionskin --help` readable end-to-end.

No behavioral changes except: `--norm-mode` default (Priority 14.3) and `--strict-bic` argparse type (Priority 14.2).

## Priority 14.3 — `--norm-mode` behavior change

**Goal:** Change `--norm-mode` to default `None` so each pipeline can use its own built-in default when no global override is set. This is a behavior change.

**Depends on:** 14.1 (norm-mode in Universal section), 14.2 (rename pass complete)  
**Blocks:** nothing directly, but must be complete before the test suite (14.7) can be validated

### Behavior spec

| Condition | Behavior |
|---|---|
| Neither `--norm-mode` nor `--<pipeline>-norm-mode` set | Pipeline uses built-in default: growth→`chrom-median`, rms→`chrom-median`, hmm→`ref-stage` |
| `--norm-mode` set, no `--<pipeline>-norm-mode` | Pipeline inherits `--norm-mode` value |
| `--<pipeline>-norm-mode` set | That value is used regardless of `--norm-mode` |

### Implementation locations

1. **`build_parser()`** — change `--norm-mode` default from `"chrom-median"` to `None`

2. **`_resolve_pipeline_norm_mode()` at `onionskin.py:1989`** — when `shared` (the `--norm-mode` value) is `None`, `_default_pipeline_norm_mode()` must return the pipeline's built-in default instead of forwarding `None`

3. **`getattr(args, "norm_mode", "chrom-median")` at `onionskin.py:1996`** — change fallback to `None`: `getattr(args, "norm_mode", None)`

4. **`getattr(args, "norm_mode", "chrom-median")` at `onionskin.py:2215`** — same change

5. **`onionskin.py:512`** — `norm_mode = norm_mode_override or getattr(args, "norm_mode", "chrom-median")` → change fallback to `None`

Note: HMM currently defaults to `ref-stage` (not `chrom-median`). In a future phase, HMM may be updated to default to `chrom-median`. That is out of scope for Phase 14 — see `KNOWN_ISSUES.md` entry added during brainstorming.

## Priority 14.4 — Deprecation error catches

**Blocks:** 14.7 (tests must be updated to expect deprecation errors)

### Implementation pattern

```python
_DEPRECATED_FLAGS = {
    "--z-thresh-single": "--rms-z-thresh",
    "--halfwidths-kb-single": "--rms-halfwidths",
    ...
    "--strict-bic": "--shape-score-strict-bic on",  # note: includes value
    ...

def _check_deprecated_flags():
```

## Priority 14.11 — AGENT_CONVENTIONS.md CLI flag conventions rule

**Goal:** Codify the Universal + pipeline-specific override pattern so future phases can add flags without debate.

**Depends on:** nothing (independent)  
**Blocks:** nothing

### Rule to add to `multi-agent/AGENT_CONVENTIONS.md`

Add a section: **CLI Flag Naming and Structure Conventions**

Core rules:
1. If a flag was previously restricted to one pipeline but now applies to more, add a canonical flag in the Universal section plus pipeline-specific override flags (default `None`, inheriting from the universal unless explicitly set).
2. If a new flag will eventually apply to multiple pipelines, add it to Universal immediately — even if only one pipeline implements it yet — with a help-string note on current coverage.
3. Pipeline-specific override flags default to `None` and inherit from the universal unless set.
4. Use the `--<pipeline>-<concept>` naming pattern for pipeline-specific flags.
5. The universal flag's help string must list which `--<pipeline>-<concept>` overrides exist.
6. Units (e.g., kb) belong in help strings, not flag names.
7. Remove legacy aliases from the parser via a pre-parse `sys.argv` scan — do not leave SUPPRESS entries in argparse.
8. `onionskin_core/` function signatures are internal — do not rename internal parameters to match CLI renames. Conversion between CLI types (e.g., `"on"/"off"` strings) and internal types (e.g., `bool`) happens at the `onionskin.py` call boundary.
