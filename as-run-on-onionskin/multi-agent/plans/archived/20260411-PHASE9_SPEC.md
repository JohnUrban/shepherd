## What Phase 9 is

Phase 9 makes the HMM pipeline self-sufficient — a full successor to PuffStep. When a user
runs `--pipeline hmm`, they get:

1. Every analysis that multistage growth mode provides, adapted to use HMM-defined amplicon
   regions and boundaries.
2. HMM-specific analyses that the state path uniquely enables: nesting-level geometry, fork
   travel trajectories, developmental progression, origin stability, and multi-stage-aware
   filtering.

**Phase 10 (cross-pipeline synthesis) is NOT Phase 9.** Do not build unified result tables
that merge HMM outputs with multistage growth outputs. That belongs in Phase 10. Phase 9
treats HMM results as standalone.

## Priority 9.1 — HMM-derived replication metrics ✓ DONE (v0.9.00)

**Module:** `onionskin_core/hmm_metrics.py` (implemented)

### Inputs (from existing step outputs)

- Step 7 (`07-collapsedHMM/`) — collapsed BED files with state labels per amplicon region
- Step 8 (`08-aboveBackground/`) — above-background regions (state > 1)
- Step 9 (`09-summitStates/`) — summit state regions (highest state per amplicon)
- Step 10 (`10-summitBins/`) — summit bin coordinates

The collapsed HMM BED (step 7) is the primary input for nesting-level extraction. Each row
is a contiguous run of a single state. Columns: chrom, start, end, state_value (integer).

### Core function: `extract_nesting_levels()`

```python
def extract_nesting_levels(
    amplicon_regions: List[Tuple[str, int, int]],  # (chrom, start, end) from step8
    collapsed_hmm_path: str,                        # step7 collapsed BED
) -> List[AmpliconLevels]:
    """
    For each amplicon region, extract all nesting levels from the HMM state path.

    A nesting level is defined by a contiguous block of a given state value that
    falls within the amplicon region. Levels are indexed from 0 = innermost
    (highest state value) to N-1 = outermost (lowest non-background state).

    Returns one AmpliconLevels per amplicon.
    """
```

### Data structures

```python
@dataclass
class NestingLevel:
    level_idx: int          # 0 = innermost (highest state), N-1 = outermost
    state_value: int        # raw HMM state integer (e.g. 3 = 8x)
    left_bp: int            # left boundary of this level (genomic coord)
    right_bp: int           # right boundary of this level
    domain_length_bp: int   # right_bp - left_bp

@dataclass
class AmpliconLevels:
    chrom: str
    amplicon_start: int     # from step8 above-background region
    amplicon_end: int
    stage: str              # stage label (e.g. "S2", "L3", "P")
    n_levels: int           # number of nesting levels
    levels: List[NestingLevel]      # ordered innermost → outermost
    summit_state_left_bp: int       # from step9 summit state region
    summit_state_right_bp: int
    origin_bp: int                  # center of summit state interval (default estimate)
    active_window_left_bp: int      # outermost level left boundary (= amplicon_start)
    active_window_right_bp: int     # outermost level right boundary (= amplicon_end)
    complexity_score: int           # = n_levels (convenient alias)
```

### Output: `11-amplicon-metrics/`

Write one TSV per stage:
`{stage}.hmm_amplicon_metrics.tsv`

Columns:
```
amplicon_id  chrom  amplicon_start  amplicon_end  stage
n_levels  summit_state_left_bp  summit_state_right_bp  origin_bp
active_window_left_bp  active_window_right_bp
level_0_state  level_0_left_bp  level_0_right_bp  level_0_domain_length_bp
level_1_state  level_1_left_bp  level_1_right_bp  level_1_domain_length_bp
...  (columns for each level, up to max_levels across all amplicons; NaN if level absent)
```

`amplicon_id` = `{chrom}:{amplicon_start}-{amplicon_end}` (consistent with step8 output).

### What NOT to do in 9.1
- Do NOT use signal thresholds to define nesting levels. Levels come from HMM state indices.
- Do NOT use the center of the full amplicon window as the origin. Use the summit-state
  interval center (step 9 output).
- Do NOT conflate step 8 (above-background) and step 9 (summit state). They are different.
  Step 8 gives the amplicon boundary; step 9 gives the summit region.

## Priority 9.2 — Fork travel and asymmetry

**New module:** `onionskin_core/hmm_fork_travel.py`

### Core concept (read this carefully)

The origin position (center of summit-state interval) is the anchor. For each nesting level:
- **Left fork travel** = origin_bp − level.left_bp
- **Right fork travel** = level.right_bp − origin_bp
- **Asymmetry** = right_travel − left_travel (signed; positive = rightward bias)
- **Asymmetry fraction** = |asymmetry| / (left_travel + right_travel)
- **Incremental travel (left)** = level[L].left_bp − level[L-1].left_bp
  (how much further left this older level's forks went vs. the next-newer level)
- **Incremental travel (right)** = level[L].right_bp − level[L-1].right_bp

Levels are indexed 0 = innermost (newest forks), N-1 = outermost (oldest forks).
Incremental travel is only defined for L ≥ 1 (comparing to the level above, L-1).

> **Sign convention note (implemented in v0.9.02):** `incremental_left_bp` is a signed
> coordinate difference. For a properly nested amplicon, outer left boundaries have smaller
> genomic coordinates than inner ones, so `incremental_left_bp` will be **negative**
> (e.g. outer left = 80, inner left = 120 → incremental = −40). `incremental_right_bp`
> will be **positive** (outer extends further right). Use `abs()` when the quantity of
> interest is "additional distance traveled" rather than direction. Notebook authors
> consuming `fork_travel_metrics.tsv` must account for this.

### Cross-stage trajectory

After computing per-stage fork travel for all stages, build the cross-stage table:

```
amplicon_id  level_idx  stage  origin_bp  left_fork_bp  right_fork_bp
left_travel_bp  right_travel_bp  asymmetry_bp  asymmetry_fraction
incremental_left_bp  incremental_right_bp
```

One row per (amplicon, level, stage). NaN when a level is absent in a stage.

### Additional per-amplicon cross-stage metrics

- **Level emergence stage:** first stage in which each level_idx appears.
- **Ghost level flag:** boolean — level present in stage N, absent in stage N+1 (possible
  model artifact; flag for inspection, do not silently drop).
- **Fork travel growth:** does left_travel and/or right_travel increase monotonically across
  stages for a given level? Compute boolean + max delta.

### Visualizations (write to `12-fork-travel/plots/`)

1. **Fork travel trajectory plot** (per amplicon):
   - Y-axis: stages (stage 1 at top, stage N at bottom)
   - X-axis: genomic distance from origin (negative = left, positive = right)
   - Each level = distinct color; left and right fork positions as points per stage
   - Save as `{amplicon_id}.fork_travel_trajectory.png`

2. **Asymmetry scatter** (all amplicons, all stages):
   - X-axis: left fork travel (bp); Y-axis: right fork travel (bp)
   - Diagonal line = symmetric fork travel
   - Color by stage; one point per (amplicon, level, stage)
   - Save as `all_amplicons.fork_asymmetry_scatter.png`

3. **Level emergence heatmap**:
   - Rows: amplicons; Columns: stages; Color: n_levels active
   - Save as `all_amplicons.level_emergence_heatmap.png`

4. **Nested domain diagram** (per amplicon, per stage):
   - Stacked nested rectangles; each level a different color
   - Save as `{amplicon_id}.{stage}.nested_domains.png`

Use matplotlib. Follow the style conventions in `onionskin_core/profile_plots.py` and
`onionskin_core/aps_plots.py` for font sizes, figure sizing, and file saving patterns.

## Priority 9.3 — Multi-stage-aware filtering and summit refinement

### Multi-stage-aware filtering (critical — read carefully)

**The filter must be applied AFTER collecting evidence from ALL stages.**

```python
def filter_amplicons_multistage_aware(
    amplicon_levels_by_stage: Dict[str, List[AmpliconLevels]],
    min_width_bp: int = 0,          # 0 = no filter
    require_multistage_growth: bool = False,
) -> Dict[str, List[AmpliconLevels]]:
    """
    Filter amplicons using cross-stage evidence.

    An amplicon that fails min_width_bp in an early stage is RETAINED if:
      - It appears in multiple stages AND shows fork travel growth OR new nesting
        levels in later stages.
    Only applies width filter to amplicons seen in exactly one stage with no growth.
    """
```

**Never apply per-stage width filtering before this function.** If step 8 already applied
a hard width filter via `--hmm-min-width`, that happens before Phase 9 and is the user's
choice. Phase 9 filtering adds a second, cross-stage-aware pass on top.

### Summit refinement across stages

Three strategies, all implemented:

```python
def refine_summit_multistage(
    amplicon_levels_by_stage: Dict[str, List[AmpliconLevels]],
    strategy: str = "coverage",   # "narrowest" | "coverage" | "combined"
) -> Dict[str, Tuple[int, int]]:
    """
    Returns {amplicon_id: (refined_summit_left_bp, refined_summit_right_bp)}.

    Strategies:
      "narrowest"  — smallest summit-state interval across all stages
      "coverage"   — genomic coverage depth of summit-state intervals across stages;
                     return central region with highest cross-stage overlap count
      "combined"   — intersection of narrowest and coverage results; fallback to
                     coverage if intersection is empty
    """
```

Also compute:
- **Permissive summit:** union of all summit-state intervals across stages (widest estimate)
- **Origin stability:** stddev of origin_bp across stages for each amplicon. High stddev
  may indicate noisy estimation or genuine biological shift.

Output: `13-summit-refinement/{trajectory_id}.summit_refinement.tsv`
and `13-summit-refinement/all_trajectories.summit_refinement.tsv`
Columns: `trajectory_id  amplicon_id  chrom  amplicon_start  amplicon_end  stage_count
permissive_left  permissive_right  narrowest_left  narrowest_right
coverage_left  coverage_right  combined_left  combined_right
origin_stddev_bp  filter_keep  multistage_growth_evidence`

> **Schema note (implemented in v0.9.04):** In step-13 outputs, `amplicon_id` is set equal
> to `trajectory_id`. These are cross-stage merged records, not single-stage amplicon calls.
> The `amplicon_id` column here does NOT correspond to any individual stage-11 `amplicon_id`.
> Notebook authors should use `trajectory_id` as the join key to step-11/step-12 data.

## Open questions and known artifacts

### Right-side asymmetry bias in fork travel

During initial plot review (v0.9.07 outputs), a systematic bias was observed where
`right_travel_bp > left_travel_bp` across amplicons, suggesting the HMM consistently
calls the right boundary of an amplicon domain further from the origin than the left.

**Suspected cause — directional Viterbi artifact:**
The Viterbi decoder processes bins left to right. On the left side of an origin, coverage
rises from background into signal and the decoder transitions up (no prior high-state
inertia). On the right side, coverage falls back toward background, but the decoder has
been accumulating high-state history — the transition penalty holds it in the high state
slightly longer than symmetry would predict. Net result: the right domain boundary is
systematically placed slightly further from the origin than the left boundary, inflating
`right_travel_bp` relative to `left_travel_bp` for every amplicon.

**Why this might not be a simple code fix:**
Reversing the Viterbi direction (right to left) would flip the bias rather than remove it.
A bidirectional decode (forward + backward, take intersection or average) would be the
principled fix but is a significant algorithm change.

**Short-term approach:**
- Document the bias in plot outputs (add annotation to fork travel plots)
- Record per-amplicon asymmetry in step-12 outputs (already done via `asymmetry_fraction`)
- Do NOT attempt to correct it within the HMM until Phase 10 cross-pipeline data is available

**Phase 10 calibration path:**
Compare HMM asymmetry values to multistage-gradient and per-stage mean-shift boundary
estimates for the same amplicons. If all three show similar asymmetry, it is likely real
biology (e.g., leading vs. lagging strand differences, genomic orientation effects). If
only HMM shows it, it confirms the Viterbi artifact hypothesis, and the other pipeline
estimates can serve as a correction reference.

> **ACTION NEEDED (before 9.5/10):** In the fork travel plots, annotate the mean asymmetry
> direction so users are aware of the potential bias. Consider adding a warning to the
> `hmm_state_path_qc.ipynb` notebook if the dataset-wide median `asymmetry_fraction`
> exceeds a threshold (e.g., 0.1 in absolute value) in one direction.

#### Plot A — Fork Asymmetry Scatter (`all_amplicons.fork_asymmetry_scatter.png`)

**Status: KEEP with fixes.**

**What it shows:** One point per (amplicon, stage, level). X = left fork travel (bp),
Y = right fork travel (bp). Diagonal = perfect symmetry. Points above diagonal = right-biased.

**Findings from review:**
- The plot looks reasonable — there is more right-side asymmetry than left but also genuine
  left-side asymmetry; it is not a purely one-sided artifact.
- Stage 5 dots dominate and obscure earlier stages visually.
- Axis orientation is counter-intuitive: left fork travel on X (which runs to the right)
  and right fork travel on Y (which runs up, on the left of the plot). Should be swapped:
  **right fork travel on X, left fork travel on Y.**
- Symmetry concern: if the origin anchor used is the center of the summit-state interval,
  that may artificially inflate apparent symmetry because both forks are measured from the
  center of the same interval they define. If the true biological origin is off-center, this
  introduces a systematic error. Worth noting in the plot and revisiting once Phase 10
  provides independent summit estimates.

**Action items:**
1. **Swap axes:** right_travel_bp → X axis, left_travel_bp → Y axis.
2. **Add per-stage grid variant:** a grid of scatter plots, one panel per stage, so
   earlier stages are not obscured. This is a supplemental/diagnostic figure; implement
   alongside the existing combined scatter (do not replace it).
3. **Add note to plot title or caption** about the symmetry artifact risk when origin = summit
   center. (Can be done as a subtitle or annotation, not a code-level fix.)

#### Plot C — Fork Travel Trajectory (`{trajectory_id}.fork_travel_trajectory.png`)

**Status: KEEP with terminology clarified; separate fork-age plot added.**

**What it currently shows:**
- Y axis = stage (time runs downward, stage_1 at top).
- X axis = distance from origin in bp: left fork travel as negative, right as positive,
  origin at x=0.
- Each colored line = one stage-rank nesting level label (L0, L1, ...) across stages.
- **Crucially: L0 means "the innermost/newest level IN THAT STAGE."** This means L0 in
  stage 2 and L0 in stage 5 refer to different biological fork sets (the newest set in each
  respective stage). Similarly L1 = second-newest in each stage.

**Why this is not what was originally intended:**
The user wanted to track the SAME biological fork set across stages — i.e., follow the
first-ever-fired forks (always at the outermost boundary), then the second-fired forks
(next layer inward), etc. In the current plot, the "oldest" forks are always at the highest
level index (e.g., L2 in a 3-level amplicon), but that index is not stable — a stage with
only 1 level has only L0, so the oldest forks disappear as a distinct series rather than
being tracked.

**Why it is still useful:**
- Visually shows how the summit-state interval (L0 in each stage) moves and narrows/widens
  across stages — useful for summit stability diagnostics.
- Shows how many nesting levels emerge per stage at a glance.
- Provides a cross-stage view of the HMM's raw output that complements the static
  nested-domain diagrams.

**Terminology issue:** "L0", "L1" etc. is not intuitive because the biological meaning
depends on context (it is a rank within a stage, not a persistent fork identity). This is
now clarified in plot captions/titles and in the HMM fork-travel notebook: L0/L1 are kept
as stage-rank labels, while fork-age plots carry the cross-stage biological identity.

**New plot needed — Fork Age Tracking:**
A separate plot that tracks the SAME fork set across stages, identified by their "age"
(firing order). Key insight: the oldest forks (first re-replication round) are always at
the outermost boundaries of the amplicon in every stage. The second-oldest are always the
next layer inward. This ordering is stable and consistent across stages.

Mapping: in a stage with N nesting levels (0=innermost, N-1=outermost), fork age index
`a` corresponds to level index `N-1-a`. So:
- Fork age 0 (oldest, outermost): level index `N-1` in each stage
- Fork age 1 (second oldest): level index `N-2`
- Fork age k (newest visible in this stage): level index `0` (= current L0)

Only fork ages that are visible in ALL (or most) stages can be meaningfully plotted as
continuous trajectories. Stages with fewer levels than others will simply not have entries
for the younger fork ages. This is expected and should be rendered as a gap in the line,
not an error.

This "fork age tracking" plot is a NEW addition, not a replacement. Both plots serve
different diagnostic purposes. Implemented in v0.9.12 using a derived step-12 table,
`fork_age_metrics.tsv`, with missing-stage gaps rendered explicitly.

## Coding conventions (must follow)

1. **No new external dependencies.** All Phase 9 code must work with the packages already
   in the project (numpy, pandas, scipy, matplotlib, sklearn). Do not add new pip deps.

2. **Parallel execution.** Where per-amplicon or per-stage processing is independent, use
   `concurrent.futures.ThreadPoolExecutor` (matching the pattern in `hmm_engine.py`).

3. **Atomic writes.** Write to a `.tmp` file, then `os.replace(tmp, final)`. See how
   `_step9_summit_states()` does this.

4. **Logging.** Use `from onionskin_core.log import log` and `log.verbose(...)`.
   Do not use `print()` for operational messages.

5. **Output directory creation.** `os.makedirs(outdir, exist_ok=True)` at the top of each
   step function.

6. **Wiring.** New Phase 9 steps are called from `run_hmm()` in `hmm_engine.py`. Add them
   after step 10. Each step gets its own clearly-named subdirectory in the output layout.
   Add the new directory keys to the `d` dict at the top of `run_hmm()`.

7. **Schema consistency.** `amplicon_id` must be `{chrom}:{amplicon_start}-{amplicon_end}`
   throughout all Phase 9 outputs. Stage labels must match the existing convention used
   elsewhere in onionskin (check `multistage_engine.py` for the canonical stage label format).

8. **Do not modify existing steps 1–10.** Phase 9 adds new steps downstream of step 10.
   Existing step outputs are inputs only.

## Validation expectations

- **Priority 9.1:** For each amplicon in the gold-standard dataset, manually verify that
  nesting levels match what is visible in the state path (inspect step 7 output). The
  summit-state interval should match step 9 output exactly.
- **Priority 9.2:** Fork travel distances for the innermost level should be smaller than
  for the outermost level (nested structure means inner < outer). If outer < inner for any
  level, something is wrong.
- **Priority 9.3:** The "narrowest" refined summit should always be ≤ width of the
  "permissive" (union) summit. If narrowest > permissive, something is wrong.
- **Priority 9.4:** APS values from HMM-defined loci should be broadly comparable (though
  not identical) to APS values from multistage growth loci for the same amplicons.
